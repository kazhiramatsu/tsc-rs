How to use / maintain:

- This is a NAVIGATION and review aid. The port spec remains the tsc
  source itself (README prime directive): transcribe from the anchor,
  never from these summaries.
- The join key is the FUNCTION NAME. Line anchors drift on re-vendor;
  the inventory regenerates mechanically, this file is hand-maintained
  — on re-vendor, re-audit only rows whose function bodies changed.
- Module home and NAMED/helper status are NOT duplicated here; read
  them from the inventory row of the same name.
- Codes columns here EXPAND the inventory's `…`-truncated lists to
  the full sets sighted in source.

Corrections and artifacts this audit surfaced (feed these into the
`xtask codegen band-inventory --by-function` tool spec):

- `getResolvedSignatureWorker` @47312 is a nearest-enclosing-function
  misattribution (confirming impl-checker-2xxx.md §11's caveat about
  this anchor): the line lands on the `syncIterationTypesResolver` /
  `asyncIterationTypesResolver` DATA OBJECTS at `createTypeChecker`
  scope. The six iterator-shape codes are emitted by the
  `getIterationTypesOf*` machinery that consumes those resolver
  objects; the real `getResolvedSignatureWorker` (46981) emits none
  of them.
- `visit` @49912 merges two same-named nested functions: 2308 is in
  `getExportsOfModuleWorker`'s visit; 2744 is in a different nested
  `visit` inside `checkTypeParametersNotReferenced` (@84863).
- `elaborateArrayLiteral` @64449: the anchor lands in
  `generateObjectLiteralElements`; `elaborateArrayLiteral` itself
  (64410) emits nothing — the 2418 site belongs to the generator
  feeding the object-literal elaboration.
- Two rows count RELATED-INFO attachments as emissions:
  `checkSpreadPropOverrides` 2785 is related info under primary 2783,
  and the 2843 site @83190 is related info under primary 2842
  (`checkPotentialUncheckedRenamedBindingElementsInTypes`).
- `reportImplementationExpectedError` (12 codes): only
